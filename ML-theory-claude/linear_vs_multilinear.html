Simple Linear vs Multiple Linear Regression
Two models, same family — the difference is how many features you use to make predictions
Simple Linear Regression — 1 feature
Predicts a continuous output using exactly one input feature. Draws the best straight line through the data.

Formula:
y = w₁x + b y = prediction (what we output)
x = the single input feature
w₁ = slope (how much y changes per unit of x)
b = intercept (value of y when x = 0)

Real example: Predict house price from size only.
Price = 150 × size + 20000
Multiple Linear Regression — many features
Predicts a continuous output using two or more input features. Fits a hyperplane (flat surface) in high-dimensional space.

Formula:
y = w₁x₁ + w₂x₂ + ... + wₙxₙ + b x₁, x₂, ... xₙ = multiple input features
w₁, w₂, ... wₙ = one weight per feature
b = intercept / bias term

Real example: Predict house price from size, bedrooms, age, location score.
Price = 150×size + 8000×bedrooms − 500×age + 200×location + 5000
The key insight: Simple linear regression is just multiple linear regression with n=1. They use the same math, same loss function (MSE), same optimisation (gradient descent or normal equation). The only difference is how many features are in the equation.
What both share
· Linear relationship between features and output
· Same loss function: MSE
· Same optimisation: gradient descent
· Coefficients interpretable as feature weights
· Same assumptions (linearity, normality of errors)
· Output is always a continuous number
Simple LR — when to use
· Quick baseline model
· Understand one key driver
· Very small datasets
· When one feature dominates
· Teaching / explanation

1 feature only Easy to visualise
Multiple LR — when to use
· Real-world problems (always multiple factors)
· Better accuracy than simple LR

The geometry — what each one draws
Simple LR — draws a LINE
With 1 feature (x axis) and 1 output (y axis), the model is a 2D space. The best fit is a straight line.

You can visualise it perfectly on paper.

2 parameters: slope w₁ and intercept b

The line minimises the sum of squared vertical distances from each data point to the line (MSE).
Multiple LR — draws a HYPERPLANE
With 2 features it's a plane in 3D space. With 3+ features it's a hyperplane — impossible to visualise but the math is identical.

n+1 parameters: one weight per feature + intercept

The hyperplane minimises the sum of squared perpendicular distances from each point to the surface.

Interpreting slope w₁
Slope = how much y changes for every 1-unit increase in x.

If w₁ = 1.5 and x = house size in m²: for every extra 1 m², price increases by 1.5 units.

Positive slope = positive relationship. Negative slope = inverse relationship.
Interpreting intercept b
Intercept = predicted value of y when x = 0.

Often not meaningful in practice (e.g. a house of 0 m² has no real price). But mathematically necessary to position the line correctly.

Always include intercept unless data is centred at 0.
What R² tells you
R² = proportion of variance in y explained by x.

R²=0.85 → x explains 85% of variation in y. The remaining 15% is due to other factors not in the model.

For simple LR, R² = correlation² between x and y.
How it is solved: Two approaches give the exact same answer.
1. Normal equation: w = (XᵀX)⁻¹Xᵀy — exact, instant, works for small data
2. Gradient descent: iteratively minimise MSE — scales to any data size (used by Spark ML)

Interpreting each coefficient
Each weight wᵢ = change in y for a 1-unit increase in xᵢ holding all other features constant.

w₁=8 means: one extra m² adds £8k to price, assuming bedrooms and age stay the same.

This "all else equal" interpretation is the key difference from simple LR.
Multicollinearity problem
When two features are highly correlated (e.g. size and number of rooms), coefficients become unstable and hard to interpret.

The model may give one feature a huge positive weight and another a huge negative weight to compensate.

Fix: Remove one of the correlated features, or use Ridge regression (L2 regularisation).
Feature scaling matters
Size ranges 50–300 m². Age ranges 0–100 years. Bedrooms ranges 1–6.

Without scaling, gradient descent updates size weights much faster than bedroom weights (larger gradient).

Fix: Always z-score normalise features before training. StandardScaler in PySpark.

Assumptions of linear regression — both types
Assumptions you must check
1. Linearity — relationship between X and y is linear. Check: residual vs fitted plot (should be random scatter).

2. Normality of residuals — errors approximately normally distributed. Check: histogram of residuals.

3. Homoscedasticity — constant variance of errors. Check: residual plot (no funnel shape).

4. No multicollinearity — features not highly correlated with each other. Check: correlation matrix, VIF score.

5. Independence — observations independent of each other. Important for time series data.
What happens when assumptions are violated
Non-linearity → Add polynomial features (x²) or use tree-based model.

Non-normal residuals → Log-transform target. Use Huber loss.

Heteroscedasticity → Log-transform target or features. Use weighted regression.

Multicollinearity → Remove correlated features. Use Ridge (L2) regularisation which shrinks collinear weights.

Autocorrelation → Add lag features. Use time series models.

Databricks + PySpark Implementation
Complete code for both regression types on Delta tables — from raw data to MLflow
Important: In Spark ML, both simple and multiple linear regression use the exact same class — LinearRegression(). The difference is purely in how many columns you pass to VectorAssembler. One column = simple. Multiple columns = multiple LR.
Simple Linear Regression in PySpark
One feature → VectorAssembler with one column → LinearRegression. Coefficient array has exactly 1 element.
Multiple Linear Regression in PySpark
Many features → VectorAssembler with multiple columns → same LinearRegression. Coefficient array has n elements — one per feature.
from pyspark.ml.regression import LinearRegression from pyspark.ml.feature import VectorAssembler, StandardScaler from pyspark.ml.evaluation import RegressionEvaluator from pyspark.ml import Pipeline import mlflow # ── Load data from Delta table ───────────────────────────────── df = spark.table("catalog.schema.house_prices") train, val, test = df.randomSplit([0.7, 0.15, 0.15], seed=42) # ══════════════════════════════════════════════════════════════ # SIMPLE LINEAR REGRESSION — one feature only # ══════════════════════════════════════════════════════════════ assembler_simple = VectorAssembler( inputCols=["size"], # ← ONE feature outputCol="features" ) lr_simple = LinearRegression( featuresCol="features", labelCol="price", regParam=0.01 # L2 regularisation ) pipeline_simple = Pipeline(stages=[assembler_simple, lr_simple]) with mlflow.start_run(run_name="simple_lr"): model_simple = pipeline_simple.fit(train) preds = model_simple.transform(val) rmse = RegressionEvaluator(metricName="rmse").evaluate(preds) r2 = RegressionEvaluator(metricName="r2").evaluate(preds) mae = RegressionEvaluator(metricName="mae").evaluate(preds) mlflow.log_param("n_features", 1) mlflow.log_metric("val_rmse", rmse) mlflow.log_metric("val_r2", r2) mlflow.log_metric("val_mae", mae) # Extract learned parameters lr_model = model_simple.stages[-1] print(f"Slope (w1): {lr_model.coefficients[0]:.4f}") print(f"Intercept (b): {lr_model.intercept:.4f}") print(f"R²: {r2:.4f}") print(f"RMSE: {rmse:.4f}") # ══════════════════════════════════════════════════════════════ # MULTIPLE LINEAR REGRESSION — many features # ══════════════════════════════════════════════════════════════ feature_cols = ["size", "bedrooms", "age", "location_score", "floor"] assembler_multi = VectorAssembler( inputCols=feature_cols, # ← MULTIPLE features outputCol="raw_features" ) # Scale features — critical for gradient descent convergence scaler = StandardScaler( inputCol="raw_features", outputCol="features", withMean=True, withStd=True ) lr_multi = LinearRegression( featuresCol="features", labelCol="price", regParam=0.1, # L2 — prevents multicollinearity issues elasticNetParam=0.0 # 0.0 = pure Ridge ) pipeline_multi = Pipeline(stages=[assembler_multi, scaler, lr_multi]) with mlflow.start_run(run_name="multiple_lr"): model_multi = pipeline_multi.fit(train) preds = model_multi.transform(val) rmse = RegressionEvaluator(metricName="rmse").evaluate(preds) r2 = RegressionEvaluator(metricName="r2").evaluate(preds) mlflow.log_param("n_features", len(feature_cols)) mlflow.log_metric("val_rmse", rmse) mlflow.log_metric("val_r2", r2) # Extract and interpret coefficients lr_model = model_multi.stages[-1] coefs = lr_model.coefficients for feat, coef in zip(feature_cols, coefs): print(f"{feat:20s}: {coef:+.4f}") mlflow.log_param(f"coef_{feat}", round(float(coef), 4)) print(f"Intercept: {lr_model.intercept:.4f}") # 1. Check multicollinearity — correlation matrix from pyspark.ml.stat import Correlation vec_df = assembler_multi.transform(df) corr_matrix = Correlation.corr(vec_df, "raw_features") print("Correlation matrix:"); print(corr_matrix.head()[0].toArray().round(2)) # 2. When to use which in practice
Use Simple LR when
· Building a quick baseline
· Explaining one key driver to stakeholders
· Only one feature is available or relevant
· You want to visualise the relationship
· EDA — understand individual feature impact

Always start here for regression
Use Multiple LR when
· Simple LR underfits (R² too low)
· Multiple factors clearly drive the target
· You need to control for confounders
· Your Delta table has multiple useful features
· Interpretability is required (e.g. finance)

The standard production regression model
Move beyond both when
· Residual plot shows curves (non-linearity)
· R² stuck below 0.6 despite many features
· Interactions between features matter (x₁×x₂)
→ Add polynomial features
→ Try Random Forest or XGBoost

Tree models handle non-linearity automatically
