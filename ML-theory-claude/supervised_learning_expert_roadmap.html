The 80/20 rule for tabular data
For tabular data (your Delta tables), these 4 algorithms cover 95% of real-world problems:

1. Logistic/Linear Regression — baseline always
2. Random Forest — robust default
3. XGBoost / LightGBM — production winner
4. MLP — when above 3 plateau

Master these 4 before touching anything else.
The 4 families and their logic
Linear: draws a line/hyperplane. Fast, interpretable. Fails on curves.

Tree-based: asks yes/no questions. Handles any shape. No scaling needed.

Distance-based: similar items are close. Needs scaling. Slow at prediction.

Probabilistic: models class distributions. Fast, needs distribution assumptions.
Default algorithm by problem
Tabular classification: XGBoost
Tabular regression: XGBoost
Text classification: Logistic + TF-IDF
Image classification: CNN (PyTorch)
Interpretability required: Logistic/Decision Tree
Streaming data: SGDClassifier
Small dataset: SVM or LDA
Time series: XGBoost + lag features
RIGHT SITUATION → RIGHT ALGORITHM
Which Algorithm for Which Situation
Real-world scenarios mapped to the best algorithm choice
The most important skill in ML engineering is not tuning hyperparameters — it is matching the algorithm to the problem. Wrong algorithm choice means months of wasted effort. This tab is your decision guide for every real situation.
Situation decision cards
You need to predict a continuous number (price, temperature, demand)
Start with Linear Regression as baseline. If R²<0.7, move to Random Forest. If still not enough, use XGBoost. Only move to Neural Net if XGBoost fails on complex feature interactions.
→ Linear Regression → Random Forest → XGBoost → MLP
Binary classification — churn, fraud, click prediction
Start with Logistic Regression (fast baseline, interpretable). If AUC<0.80, try Random Forest. If still struggling, XGBoost. Add class_weight if imbalanced.
→ Logistic Regression → Random Forest → XGBoost
Multi-class classification — product category, species, document topic
Logistic Regression handles multiclass natively (multinomial). Random Forest and XGBoost also handle it directly. For text use Naive Bayes or SGD+TF-IDF first.
→ Logistic Regression (multinomial) → Random Forest → XGBoost
Interpretability is mandatory (finance, healthcare, legal)
Must use white-box models. Decision Tree for rule extraction. Logistic Regression for probability + coefficients. Add SHAP to XGBoost for post-hoc explanations if accuracy demands it.
→ Logistic Regression → Decision Tree (max_depth=5) → XGBoost + SHAP
Very small dataset (less than 1000 rows)
Deep learning is impossible. Tree ensembles may overfit. Use: SVM (very effective on small data), LDA/QDA (if features are Gaussian), Logistic with L2 reg, or KNN. Avoid deep trees.
→ SVM → Logistic Ridge → LDA → KNN (k=5-10)
Very large dataset (millions of rows, Spark/Databricks)
Single-machine algorithms (SVM, KNN, GP) are too slow. Use distributed Spark ML: LinearRegression or LogisticRegression (fast, scales perfectly), then GBTClassifier for accuracy.
→ Spark LinearRegression → Spark LogisticRegression → Spark GBT
Streaming data / online learning (partial_fit)
Only SGDClassifier and SGDRegressor support online learning with partial_fit() in sklearn. Naive Bayes also supports it. These are the only options for truly streaming pipelines.
→ SGDClassifier(loss='log_loss') → SGDRegressor → NaiveBayes
Text classification (spam, sentiment, topic)
Start with TF-IDF + Logistic Regression — surprisingly powerful and fast. Naive Bayes (Multinomial or Complement) is another strong baseline. SGD scales to millions of documents.
→ TF-IDF + LogReg → Complement NB → SGD → fine-tuned BERT
Severe class imbalance (fraud, rare disease — less than 5% minority)
Algorithm matters less than strategy. Add class_weight='balanced' to any model. Use SMOTE for oversampling. For extreme imbalance, consider Isolation Forest (anomaly detection) instead of classification.
→ XGBoost(scale_pos_weight) → LogReg(class_weight) → Isolation Forest
Features have non-linear relationships with target
Linear models will underfit. Tree-based models (Random Forest, XGBoost) capture non-linearity automatically through splits. SVM with RBF kernel also works. Do NOT use Linear Regression without polynomial features.
→ Random Forest → XGBoost → SVM (RBF kernel) → MLP
Many correlated features (multicollinearity)
Logistic/Linear regression with collinear features gives unstable coefficients. Use Ridge (L2) which handles multicollinearity well. Tree-based models are completely immune to multicollinearity.
→ Ridge Regression → Lasso (feature selection) → Random Forest
Mixed data types (numbers + categories + text + dates)
XGBoost and LightGBM handle mixed types natively (can use categorical encoding). Random Forest also works. Linear models need manual encoding of all categorical features first.
→ LightGBM (native categoricals) → XGBoost → Random Forest
Probability calibration needed (risk scoring, decision threshold)
Logistic Regression is naturally calibrated. Random Forest and XGBoost have good AUC but poorly calibrated probabilities — always apply CalibratedClassifierCV after training if you use probabilities for decisions.
→ Logistic Regression → RF/XGBoost + CalibratedClassifierCV
Fast inference needed (real-time API, <10ms)
Linear models predict in microseconds (single dot product). Shallow trees also fast. Deep ensembles (1000 trees) or neural nets are slower. For real-time serving prefer LogReg or a small XGBoost model.
→ Logistic Regression → Decision Tree (depth≤8) → Small XGBoost
High-dimensional data (many features, sparse)
Sparse high-dim data (TF-IDF vectors, one-hot encoded) works best with linear models (LogReg with L1, Linear SVM). Tree ensembles struggle with very sparse data. SGD is ideal for large sparse data.
→ LogReg (L1) → LinearSVC → SGDClassifier → Naive Bayes
Time series with supervised framing
Create lag features, rolling statistics, time-based features. Then XGBoost/LightGBM with those features beats most time-series-specific models. Never use KNN or SVM on raw time series.
→ XGBoost + lag features → LightGBM → Linear with time features

WHERE TO START
Complete Beginner Path — From Zero to First ML Model
The exact sequence a beginner should follow — no skipping ahead
The biggest mistake beginners make: jumping to XGBoost or Neural Networks without understanding why simpler models fail first. You must understand WHY a model fails before you know which model to try next. The sequence below builds this intuition step by step.
Phase 1 — Foundations (Week 1–2)
1
Linear Regression — the foundation of everything
Every ML concept (loss function, gradient descent, overfitting, coefficients) appears in its simplest form here. Master this before anything else.
✓ Predict house prices with 1 feature (simple LR)
✓ Add more features (multiple LR)
✓ Plot residuals, compute R², RMSE
✓ See what happens when you add irrelevant features
✓ Understand: w = (XᵀX)⁻¹Xᵀy
2
Logistic Regression — classification fundamentals
Binary classification, probabilities, sigmoid function, decision boundary, confusion matrix. This is the gateway to classification ML.
✓ Predict survival (Titanic) — binary classification
✓ Understand sigmoid: p = 1/(1+e^(-z))
✓ Plot confusion matrix, compute precision/recall
✓ Tune C (regularisation) — watch bias/variance change
✓ Visualise the decision boundary
Phase 2 — Tree Models (Week 3–4)
3
Decision Tree — understand splits and overfitting visually
The best model for building intuition about overfitting and bias-variance. You can see the tree and understand every decision.
✓ Train a deep tree — watch it perfectly memorise training data
✓ Print the tree — read the actual rules
✓ Increase max_depth from 1 to 20 — plot train vs val accuracy
✓ Find the optimal depth — your first hyperparameter search
✓ Extract feature importances
4
Random Forest — your first ensemble
Shows how combining weak models creates something stronger. Introduces bagging, OOB score, and feature importance in a more reliable form.
✓ Compare single tree vs Random Forest on same data
✓ Use OOB score as a free validation set
✓ Plot feature importances as a bar chart
✓ Tune n_estimators and max_depth
✓ See how variance decreases vs single tree
Phase 3 — The Production Winner (Week 5–6)
5
XGBoost / LightGBM — the industry standard
This is the algorithm that wins most Kaggle competitions on tabular data. Understand boosting: each tree corrects the errors of the previous trees.
✓ Train XGBoost on same dataset as RF — compare AUC
✓ Use early stopping — understand why it prevents overfitting
✓ Tune learning_rate and n_estimators together
✓ Use SHAP to explain individual predictions
✓ Understand: new_tree = gradient of loss function of current ensemble
Phase 4 — Breadth (Week 7–8)
6
SVM, KNN, Naive Bayes — know when each wins
You will rarely use these over XGBoost in production, but you need to understand WHEN they are better — small data (SVM), text (NB), fast baseline (KNN).
✓ SVM on small dataset — compare to LR and RF
✓ Naive Bayes on text classification — TF-IDF + NB pipeline
✓ KNN — vary k from 1 to 50, plot accuracy vs k
✓ Understand kernel trick intuitively
✓ Know when each beats XGBoost
Your first complete project
Project template — start here before anything else
1. Pick ONE dataset from Kaggle (Titanic, House Prices, or Heart Disease)
2. Do complete EDA — understand every feature
3. Train ALL 6 algorithms above on the SAME data
4. Compare them in ONE table: AUC, RMSE, train time
5. For the best model: plot learning curve, confusion matrix, SHAP
6. Write down WHY each model performed the way it did

This single project teaches more than 10 separate tutorial datasets.
The learning trap to avoid: watching tutorials and understanding code without writing it yourself. The only way to genuinely learn ML algorithms is to train them, break them, fix them, and explain WHY they work on paper. After every model, ask: "What would happen if I doubled the regularisation? What if I removed the 3 most important features? What if the dataset was 10x larger?"

· Can look at a new dataset and immediately know which 2–3 algorithms to try and why — without running any code
· Knows the mathematical derivation of every algorithm in this guide
· Has hit every failure mode (overfitting, leakage, distribution shift, concept drift) and knows the fix from experience

· Builds end-to-end ML systems: data ingestion → feature store → model training → serving → monitoring
· Contributes to open-source ML projects
· Can mentor others through their ML progression
· Has at least one production model that processes real business data and produces real value
Weekly practice schedule (for any level)
Monday + Tuesday — Theory
Study one algorithm deeply. Read the original paper or a textbook chapter. Derive the math on paper. Understand: what does this algorithm optimise? What are its assumptions? When does it fail?
Wednesday + Thursday — Code
Implement the algorithm on a real dataset. Train it, evaluate it, tune it. Then deliberately break it — add noise, remove features, use too little data. Understand failure modes through experience, not theory.
Friday — Compare and Reflect
Compare this week's algorithm to everything you've learned before. Where does it win? Where does it lose? What would you need to see in a dataset to choose this algorithm? Write this down.

── Helper: evaluate any model ──────────────────────────── def evaluate_clf(name, model, X_tr, y_tr, X_te, y_te, cv=5): cv_auc = cross_val_score(model, X_tr, y_tr, cv=cv, scoring='roc_auc', n_jobs=-1).mean() model.fit(X_tr, y_tr) y_pred = model.predict(X_te) y_prob = model.predict_proba(X_te)[:,1] if hasattr(model,'predict_proba') else None te_auc = roc_auc_score(y_te, y_prob) if y_prob is not None else 0 te_f1 = f1_score(y_te, y_pred) te_acc = accuracy_score(y_te, y_pred) return {'name':name,'cv_auc':cv_auc,'test_auc':te_auc,'f1':te_f1,'accuracy':te_acc,'model':model} def evaluate_reg(name, model, X_tr, y_tr, X_te, y_te, cv=5): cv_r2 = cross_val_score(model, X_tr, y_tr, cv=cv, scoring='r2', n_jobs=-1).mean() model.fit(X_tr, y_tr) y_pred = model.predict(X_te) te_r2 = r2_score(y_te, y_pred) te_rmse= np.sqrt(mean_squared_error(y_te, y_pred)) te_mae = mean_absolute_error(y_te, y_pred) return {'name':name,'cv_r2':cv_r2,'test_r2':te_r2,'rmse':te_rmse,'mae':te_mae,'model':model} #
