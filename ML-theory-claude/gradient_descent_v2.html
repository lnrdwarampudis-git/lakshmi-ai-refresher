What is Gradient Descent?
The single most important algorithm in ML — the engine behind every model that learns
One sentence: Gradient descent finds the minimum of a loss function by repeatedly moving in the direction of steepest descent — like a blindfolded person walking downhill by feeling the slope under their feet at each step.
The hill-walking analogy
Imagine you are blindfolded on hilly terrain and want to reach the lowest valley.

Your position = model weights w
Your altitude = current loss L(w)
Slope under feet = gradient ∇L
Step size = learning rate α

Each step: feel slope → step downhill → repeat until flat ground (minimum).
The mathematical core
The update rule — the only formula you need:

w = w − α × ∇L(w)
w = weights (what we optimise)
α = learning rate (step size)
∇L = gradient (which way is uphill)
− = subtract = move downhill = reduce loss

Why the gradient points uphill
The gradient ∇L is the direction of steepest INCREASE in loss. Subtracting it always moves you downhill. This is guaranteed by calculus — no guesswork.
Why small steps matter
The gradient only tells you the local slope. A giant step might overshoot the valley entirely. Small α = follow the curved surface accurately.
Convergence condition
When ∇L ≈ 0, slope is flat — you are at a minimum. Training stops. In practice: stop when val loss stops improving (early stopping).
How Gradient Descent Works — step by step
The complete forward → loss → backward → update cycle
Step 1 — Forward pass: ŷ = model(X, w) compute prediction
Step 2 — Compute loss: L = loss_fn(ŷ, y) measure how wrong
Step 3 — Backward pass: ∇L = ∂L/∂w chain rule (backprop)
Step 4 — Update: w = w − α × ∇L descend one step
Step 5 — Repeat for next batch until convergence

Forward pass
Data flows left→right. Each layer: output = activation(W·input + b). Final output = prediction. Loss measures error. No weights updated yet.
Backward pass
Error flows right→left. Chain rule computes ∂L/∂w for every weight. PyTorch tracks all ops during forward and uses them automatically in loss.backward().
Weight update
optimizer.step() applies w = w − α×∇L to ALL weights simultaneously. Adam makes it smarter by adapting α per weight. This = one iteration.
# PyTorch training loop — runs this for every batch, every epoch optimizer = torch.optim.Adam(model.parameters(), lr=1e-3) for epoch in range(100): for X_batch, y_batch in dataloader: y_pred = model(X_batch) # Step 1: forward pass loss = loss_fn(y_pred, y_batch) # Step 2: compute loss optimizer.zero_grad() # clear old gradients loss.backward() # Step 3: backprop (chain rule) optimizer.step() # Step 4: update all weights mlflow.log_metric("loss", loss.item(), step=epoch)

4 main variants — each trades off speed, noise, and gradient accuracy

Practical rule: Always start with Adam (lr=1e-3) for neural networks. For Spark ML use the default solver. You rarely need to change optimizer — Adam first, always.
# All PyTorch optimizers optim.SGD(model.parameters(), lr=0.01) # plain SGD optim.SGD(model.parameters(), lr=0.01, momentum=0.9) # + momentum optim.Adam(model.parameters(), lr=1e-3) # default choice optim.AdamW(model.parameters(), lr=1e-3, weight_decay=1e-4) # Adam + L2 reg # Learning rate scheduler — reduce lr over time scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=100) for epoch in range(100): train(...) scheduler.step() mlflow.log_metric("lr", scheduler.get_last_lr()[0], step=epoch)
Problems with Gradient Descent + Fixes
Every one of these will happen in production — knowing the fix saves days of debugging
Vanishing gradient
Gradients shrink to 0 as they flow back through layers. Early layers learn nothing.

Cause: Sigmoid/tanh in hidden layers. Deep networks.
Fix: Use ReLU. Add BatchNorm. Use residual connections (ResNet).
Exploding gradient
Gradients become huge. Loss spikes to NaN.

Cause: LR too high. RNNs with long sequences.
Fix: Gradient clipping:
nn.utils.clip_grad_norm_(model.parameters(), 1.0)
Local minima + saddle points
GD gets stuck where gradient ≈ 0 but not at global minimum.

Fix: SGD noise escapes them. Momentum powers through. Random restarts. Less of a problem than theory suggests.
Wrong learning rate
Too high → oscillates/diverges. Too low → never converges.

Fix: Start with 1e-3 for Adam. Use LR finder. Use ReduceLROnPlateau scheduler to adapt automatically.
# Fixes for common problems # Vanishing gradient → ReLU + BatchNorm nn.Sequential(nn.Linear(128,64), nn.BatchNorm1d(64), nn.ReLU()) # Exploding gradient → clip gradients loss.backward() nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0) optimizer.step() # Wrong LR → adaptive scheduler scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau( optimizer, patience=5, factor=0.5) # halve lr if stuck for 5 epochs scheduler.step(val_loss) # Local minima → SGD with momentum (noise helps escape) optim.SGD(model.parameters(), lr=0.01, momentum=0.9)
When Is Gradient Descent Useful?
Where GD is used, where it isn't, and how it maps to your Databricks + Azure stack
Short answer: GD is useful whenever you need to minimise a differentiable loss function over many parameters. That covers almost ALL of modern ML and AI. The only exceptions are problems with closed-form solutions or combinatorial (discrete) problems.
Linear + logistic regression
GD minimises MSE (linear) or cross-entropy (logistic).

When GD used: Large data — Spark MLlib always uses GD internally when you call model.fit().
When NOT: Small data → normal equation gives exact solution without iterations.

LinearRegression(maxIter=100)
Neural networks — all types
The ONLY practical way to train networks with millions of parameters. No closed-form exists.

MLPs, CNNs, RNNs, Transformers, LLMs — all trained with GD + backprop.

GPT-4 trained with GD across thousands of GPUs.

PyTorch on Databricks GPU clusters
Gradient Boosted Trees (XGBoost)
XGBoost / Spark GBT use functional gradient descent — each new tree fits the gradient (residuals) of the current ensemble. Each tree = one GD step in function space.

GBTClassifier(stepSize=0.1)
LLM pre-training + fine-tuning
Pre-training: GD on next-token prediction loss over trillions of tokens.

Fine-tuning: GD with very small lr (2e-5) to adapt to your domain. Azure OpenAI fine-tuning runs GD on your uploaded data automatically.

HuggingFace Trainer + Databricks
Embeddings + RAG
Word2Vec, sentence transformers, LLM embeddings all trained by GD. Embedding vectors are model parameters updated via GD.

When you use Azure OpenAI embeddings in RAG, those vectors were shaped by GD during pre-training.

Foundation of every RAG pipeline
When NOT to use GD
Decision trees / Random Forest → greedy splitting, not GD
K-Means clustering → Lloyd's algorithm, not GD
Naive Bayes → MLE closed form, not GD
KNN → no training at all
Small linear regression → normal equation is faster

Simpler problems → simpler solutions
Your Databricks + Azure ML stack — GD is everywhere:

model.fit(train_df) in Spark ML → GD distributed across cluster nodes
loss.backward() + optimizer.step() in PyTorch → GD updating weights
Azure OpenAI fine-tuning → GD adjusting pre-trained weights on your data
XGBoost / GBT adding a tree → one step of functional GD on residuals

GD is not optional — it IS the mechanism of learning in modern ML.
