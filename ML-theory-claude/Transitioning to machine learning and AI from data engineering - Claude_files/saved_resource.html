REINFORCEMENT LEARNING — COMPLETE GUIDE
What is Reinforcement Learning?
Learning through interaction — trial and error with a reward signal
The key difference from supervised and unsupervised learning: No labelled data. No structure to discover. Instead — an agent takes actions in an environment, receives rewards, and learns a policy that maximises cumulative reward over time. The agent learns purely from consequences of its actions.

Supervised vs RL
Supervised: given correct answer for each input. Teacher tells you the right move.

RL: only gets a reward signal after taking an action. No teacher — learns from consequences.

Example: chess engine. Supervised = given correct move every turn. RL = play games, win/lose, learn what works.

RL discovers strategies humans never programmed
Key components
State (s): current situation the agent observes
Action (a): what the agent can do
Reward (r): scalar feedback after each action
Policy (π): strategy — maps states to actions
Value (V/Q): expected cumulative future reward
Episode: one complete sequence of states/actions
Discount (γ): how much future rewards matter
Where RL is used
· Game playing (AlphaGo, OpenAI Five, StarCraft)
· Robotics (locomotion, manipulation)
· Autonomous vehicles (lane changing, parking)
· Resource management (data center cooling)
· Trading strategies (portfolio optimisation)
· Recommender systems (sequential decisions)
· LLM fine-tuning (RLHF — ChatGPT, Claude)
· Drug discovery (molecule optimisation)

RL is NOT a replacement for supervised learning. For most business problems (churn, fraud, demand forecasting) supervised learning is faster, cheaper, and more reliable. Use RL only when you have a sequential decision problem with delayed rewards and an environment to train in. CORE CONCEPTS
RL Core Concepts — The Math Behind the Loop
Markov Decision Process · Bellman Equation · Value Functions · Policy
Everything in RL is built on one foundation: the Markov Decision Process (MDP). Every RL algorithm is essentially a different way to solve the same MDP. Master these concepts and every algorithm becomes a variation on the same theme.
Markov Decision Process (MDP)
MDP = (S, A, P, R, γ)
S = State space (all possible situations)
A = Action space (all possible actions)
P(s'|s,a) = Transition probability (environment dynamics)
R(s,a,s') = Reward function
γ = Discount factor (0=myopic, 1=far-sighted)

Markov property: the future depends only on the current state, not the full history. P(sₜ₊₁|sₜ,aₜ) — the next state depends only on current state and action.
The goal — maximise expected return
Return Gₜ = sum of discounted future rewards:
Gₜ = rₜ + γ·rₜ₊₁ + γ²·rₜ₊₂ + ... = Σₖ γᵏ·rₜ₊ₖ
γ=0: only care about immediate reward
γ=0.99: care about rewards far in future

Optimal policy π* = the policy that maximises expected return from every state:
π* = argmax_π E[Gₜ | π]
Value Functions — the heart of RL
State Value Function V(s)
V^π(s) = expected return starting from state s, following policy π:
V^π(s) = E_π[Gₜ | sₜ=s] = E_π[Σₖ γᵏ·rₜ₊ₖ | sₜ=s]
Intuition: "How good is it to BE in state s?"

V*(s) = optimal value = best return possible from s
Used in: Value iteration, TD learning, Actor-Critic
Action Value Function Q(s,a)
Q^π(s,a) = expected return starting from state s, taking action a, then following π:
Q^π(s,a) = E_π[Gₜ | sₜ=s, aₜ=a]
Intuition: "How good is it to take action a in state s?"

Q*(s,a) = optimal Q-value
Used in: Q-Learning, DQN, SARSA
Policy from Q: π(s) = argmax_a Q(s,a)
Bellman Equations — recursive definition of value
Bellman Expectation Equation
Value of current state = immediate reward + discounted value of next state:
V^π(s) = Σ_a π(a|s) Σ_s' P(s'|s,a)[R(s,a,s') + γV^π(s')]
This is the key recursive identity: the value of a state equals the expected reward plus the discounted value of the next state. Every RL algorithm is solving or approximating this equation.
Bellman Optimality Equation
Optimal Q-value satisfies:
Q*(s,a) = R(s,a) + γ Σ_s' P(s'|s,a) max_a' Q*(s',a')
Q-Learning update rule (the foundation of DQN):
Q(s,a) ← Q(s,a) + α[r + γ max_a' Q(s',a') − Q(s,a)]
TD error = r + γ max Q(s',a') − Q(s,a)
This is the "prediction error" that drives all Q-learning.
Exploration vs Exploitation — the fundamental dilemma
The dilemma
Exploitation: take the action you know is best. Maximises reward now but may miss better options.

Exploration: try new actions to discover potentially better strategies. Costs reward now but may find something better.

Too much exploitation = local optimum. Too much exploration = never converges.
ε-greedy (most common)
With probability ε: explore (random action)
With probability 1-ε: exploit (best known action)

Decaying ε: start ε=1.0 (all explore), decay to ε=0.01 as training progresses.

ε = ε_min + (ε_max-ε_min)·e^(-decay·step)
Other strategies
UCB: Upper Confidence Bound — choose action with highest Q + uncertainty bonus. Systematic exploration.

Boltzmann/Softmax: sample actions proportional to Q-values. Temperature τ controls exploration.

Thompson Sampling: Bayesian — sample from posterior belief over Q-values.
ALGORITHM FAMILIES
RL Algorithm Families — The Complete Taxonomy
Model-Free vs Model-Based · Value-Based vs Policy-Based · On-Policy vs Off-Policy
The most important classification: Model-free vs Model-based. In model-free RL (most algorithms), the agent does not know P(s'|s,a) — it just experiences transitions. In model-based RL, the agent learns or is given the environment dynamics and can plan ahead.
Dimension 1 — Model-Free vs Model-Based
Model-Free (most common)
Agent does NOT learn environment dynamics. Just experiences (s, a, r, s') tuples and learns directly from them.

Pros: simpler, works when model is hard to learn, no model errors
Cons: less sample-efficient, cannot plan ahead

Algorithms: Q-Learning, SARSA, DQN, PPO, A3C, SAC, TD3
90% of practical RL is model-free
Model-Based
Agent learns or is given P(s'|s,a) and R(s,a). Can simulate trajectories and plan without real interaction.

Pros: much more sample-efficient, can plan ahead
Cons: model errors compound, hard to build accurate models

Algorithms: Dyna-Q, World Models, MuZero, AlphaZero
Used when real interactions are expensive (robotics)
Dimension 2 — Value-Based vs Policy-Based vs Actor-Critic
Value-Based
Learn Q(s,a) or V(s). Derive policy implicitly: π(s) = argmax_a Q(s,a).

Algorithms: Q-Learning, SARSA, DQN, Double DQN, Dueling DQN

Best for: discrete action spaces (games, inventory)
Bad for: continuous actions (robotics)

Cannot handle continuous actions directly
Policy-Based (Policy Gradient)
Directly optimise policy π(a|s;θ). Gradient ascent on expected return.

Algorithms: REINFORCE, TRPO, PPO

Best for: continuous action spaces, stochastic policies
Bad for: high variance (needs many samples)

Can handle continuous actions
Actor-Critic (best of both)
Actor: policy π(a|s) — decides actions
Critic: value V(s) or Q(s,a) — evaluates actions

Algorithms: A2C, A3C, SAC, TD3, PPO (hybrid)

Best for: continuous + discrete, most complex envs
Lower variance than pure policy gradient

State of the art for most problems
Dimension 3 — On-Policy vs Off-Policy
On-Policy
Learns from experience generated by the CURRENT policy. Must collect new data every update.

Algorithms: SARSA, A2C, A3C, PPO

Pros: more stable, safer for real-world deployment
Cons: less sample-efficient — cannot reuse old data

Example: PPO collects a batch of experience with current policy, updates, then throws away that data and collects fresh.
Off-Policy
Learns from experience generated by ANY policy (old or different). Can reuse data with a replay buffer.

Algorithms: Q-Learning, DQN, SAC, TD3, DDPG

Pros: more sample-efficient — reuses past experience
Cons: can be unstable, harder to tune

Experience Replay: store (s,a,r,s') in buffer, sample random batches for training — breaks correlations.

Which RL Algorithm for Which Situation
Real-world scenarios mapped to the best algorithm
Most important rule: Before choosing an RL algorithm, ask "Can I solve this with supervised or rule-based methods instead?" RL is harder to train, requires an environment, and takes much longer to converge. Only use RL when the problem genuinely requires sequential decision-making with delayed rewards.
Action space situations
Action space is discrete and small (e.g. game: up/down/left/right)
Start with Q-Learning if state space is small (tabular). If state space is large or image-based, use DQN. For best performance on Atari-like games, use Double DQN + Dueling DQN + Prioritised Replay.
→ Q-Learning (small) → DQN → Double+Dueling DQN (best)
Action space is continuous (e.g. robot joint angles, steering angle)
Value-based methods (Q-Learning, DQN) cannot handle continuous actions directly. Use SAC as your first choice — it is the most stable and sample-efficient for continuous control. TD3 is also excellent.
→ SAC (most stable) → TD3 → PPO (if on-policy preferred)
Need to work with both discrete and continuous actions
PPO handles both naturally with appropriate output heads. Use Gaussian distribution head for continuous, softmax for discrete. PPO is the most general-purpose algorithm.
→ PPO with appropriate action head
Safety is critical — agent must not take dangerous actions during training
On-policy methods are safer because they only learn from current policy. PPO with constraints (PPO-Lagrangian) or TRPO with hard constraints. Off-policy methods can use dangerous past experience.
→ PPO with safety constraints → TRPO → Constrained RL
Environment interactions are expensive (real robot, real simulator)
Off-policy methods + replay buffer are more sample-efficient. SAC reuses every experience in the buffer many times. Also consider model-based RL (Dyna-Q, MBPO) which generates synthetic rollouts.
→ SAC (best sample efficiency) → Model-Based RL → Dyna-Q
I want to train a game-playing agent from scratch
Atari/screen-based: DQN family. Board games with known rules: AlphaZero self-play. Any game without needing rules: MuZero. For fastest convergence with stable training: PPO.
→ DQN (Atari) → AlphaZero (board games) → PPO (general)
Environment type situations
State space is small and tabular (few states, few actions)
Classic Q-Learning with a Q-table. No neural networks needed. Perfect for teaching RL concepts (FrozenLake, Taxi, simple grid worlds). State and action spaces must be discrete and small.
→ Q-Learning table → SARSA → Dynamic Programming
State is high-dimensional (images, raw pixels)
Need deep neural network to process states. DQN uses CNN to process frames. PPO/SAC with CNN encoder. Always preprocess: resize to 84×84, grayscale, stack 4 frames for temporal context.
→ DQN+CNN → PPO+CNN encoder → A3C (parallel)
Multi-agent environment (multiple agents interact)
Standard RL algorithms treat other agents as part of the environment. For cooperative multi-agent: MADDPG or MAPPO. For competitive: self-play with PPO. OpenAI Five and AlphaStar used self-play.
→ Self-play PPO → MADDPG → MAPPO
Environment is partially observable (agent cannot see full state)
Standard RL assumes full observability. For partial observation (POMDP): use LSTM/GRU memory in the policy network to remember past states. Frame stacking is simple alternative.
→ PPO/SAC + LSTM memory → Frame stacking
Reward is sparse — agent rarely receives any reward
Sparse rewards make learning very slow. Use reward shaping (add intermediate rewards), curiosity-driven exploration (ICM), hindsight experience replay (HER), or hierarchical RL to break task into subtasks.
→ HER (goal-conditioned) → Curiosity (ICM) → Reward shaping → Hierarchical RL
I want the most practical RL for most problems today
PPO is the default choice for most practical RL problems. It is stable, works on discrete and continuous action spaces, has few hyperparameters to tune, and is what OpenAI uses. Stable-Baselines3 makes it easy.
→ PPO (Stable-Baselines3) → SAC (if continuous and off-policy wanted)
RLHF — RL FOR LLMs
RLHF — Reinforcement Learning from Human Feedback
How ChatGPT, Claude, and Gemini are trained — the most important RL application today
This is the most relevant RL topic for your work. RLHF is the technique that takes a pre-trained LLM (trained on next-token prediction) and aligns it to be helpful, harmless, and honest. Understanding RLHF explains WHY ChatGPT and Claude behave the way they do — and how to fine-tune LLMs on Azure OpenAI.
The 3-phase RLHF pipeline
1
Phase 1 — Supervised Fine-Tuning (SFT)
Start with a pre-trained LLM (GPT-4 base, LLaMA, etc.). Fine-tune on high-quality human demonstrations — examples of what a helpful assistant should say. This gives the model a good starting policy before RL.
Input: (prompt, ideal_response) pairs written by human labellers
Loss: Cross-entropy on ideal responses
Output: SFT model — a reasonable baseline assistant
This is standard supervised learning on (X, y) pairs
2
Phase 2 — Reward Model Training
Human labellers compare pairs of model outputs — "which response is better?" This preference data trains a reward model that learns to score any response from 0 to 1 based on helpfulness and safety.
Input: (prompt, response_A, response_B, human_preference) pairs
Train a classifier: P(A preferred over B)
Output: RM(prompt, response) → scalar reward score
This reward model replaces the human labeller during RL training
3
Phase 3 — PPO Fine-Tuning
Use PPO to optimise the SFT model to maximise the reward model score. The LLM is the policy (outputs token sequences). The reward model is the environment. A KL penalty prevents the model from drifting too far from the SFT model.
Agent = SFT LLM (policy π_θ)
State = prompt + tokens generated so far
Action = next token to generate (from vocabulary of 50k+)
Reward = reward_model(full_response) − β·KL(π_θ || π_SFT)
Algorithm = PPO (clipped surrogate objective)
Why PPO for RLHF — and why not other algorithms
Why PPO specifically?
· Stable training — clipped objective prevents large policy jumps
· Works on discrete actions (tokens from vocab)
· On-policy — uses fresh model outputs each iteration
· Well-understood, easy to implement
· KL penalty easily added to loss

PPO is the standard for RLHF
The KL penalty — critical for RLHF
Without KL penalty, PPO would optimise the reward model too aggressively and break the LLM (reward hacking).

Reward = RM(response) − β·KL(π_θ || π_SFT)
β controls the trade-off: high β = stay close to SFT. Low β = maximise reward aggressively.

This prevents the model from generating nonsense that tricks the reward model.
Newer alternatives to PPO
DPO (Direct Preference Optimisation): Skips the reward model entirely. Directly trains on (preferred, rejected) pairs. Simpler, more stable than PPO. Used in many open-source models.

GRPO: Used to train DeepSeek-R1. Group Relative Policy Optimisation — samples multiple responses and uses relative scores.

DPO is replacing PPO in many modern systems
RLHF connection to your Azure OpenAI work
Azure OpenAI fine-tuning IS RLHF (simplified)
When you fine-tune GPT-4o on Azure OpenAI with (prompt, completion) pairs:

1. Phase 1 (SFT) runs under the hood — your data is the demonstration dataset
2. Azure OpenAI applies their internal RLHF alignment on top

You don't run PPO yourself — Azure handles it. Your job is providing high-quality (prompt, ideal_completion) pairs.

Quality of your training data = quality of your fine-tuned model

Current Q values q_values = self.q_net(s).gather(1, a.unsqueeze(1)).squeeze() # Target Q values (Double DQN: use online net to SELECT, target to EVALUATE) with torch.no_grad(): next_actions = self.q_net(s_).argmax(1) next_q = self.target_net(s_).gather(1, next_actions.unsqueeze(1)).squeeze() target = r + self.gamma * next_q * (1 - done) # Huber loss (more robust than MSE) loss = nn.SmoothL1Loss()(q_values, target) self.optimizer.zero_grad() loss.backward() nn.utils.clip_grad_norm_(self.q_net.parameters(), 10) # gradient clipping self.optimizer.step() return loss.item() def update_target(self): self.target_net.load_state_dict(self.q_net.state_dict()) #
