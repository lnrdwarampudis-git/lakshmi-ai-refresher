MLE — Maximum Likelihood Estimation
Why MSE and cross-entropy exist — they are not arbitrary. They come directly from probability theory.
The big idea: MLE says — find the model parameters w that make the observed training data most probable. When you derive MLE for different data distributions, you get different loss functions. MSE and cross-entropy are not invented — they are derived from first principles.
What MLE means
Given data D = {(x₁,y₁),...,(xₙ,yₙ)}, find w that maximises:

L(w) = P(D | w) = Π P(yᵢ | xᵢ, w)
In practice, maximise log-likelihood (easier):
log L(w) = Σ log P(yᵢ | xᵢ, w)
Maximising log-likelihood = minimising negative log-likelihood = minimising loss.

The loss function IS the negative log-likelihood of the assumed data distribution.
MLE → Loss function derivation
Assume Gaussian errors → yᵢ ~ N(ŷᵢ, σ²)
log P(yᵢ|xᵢ,w) = −(yᵢ−ŷᵢ)²/2σ² + const
Maximising log L → minimising Σ(yᵢ−ŷᵢ)²
→ MSE loss!

Assume Bernoulli outcomes → yᵢ ~ Bernoulli(p)
log P(yᵢ|xᵢ,w) = yᵢlog(p) + (1−yᵢ)log(1−p)
Maximising log L → minimising −Σ[ylog(p)+(1−y)log(1−p)]
→ Binary cross-entropy!

Assume Laplace errors (heavy-tailed)
→ MAE loss!
1.0
Gaussian distribution → MSE loss. Maximising Gaussian log-likelihood = minimising (y−ŷ)²/2σ² = minimising MSE. The σ parameter determines the spread of errors the model expects. Large σ = model expects large errors = less penalty per error.
Why log-likelihood?
Products of probabilities (Π) get very small very fast — numerical underflow. Log converts the product to a sum: log(Π pᵢ) = Σ log(pᵢ). Same maximum, more stable arithmetic. This is why you see "log" everywhere in ML loss functions.
MLE in practice
When you call model.fit() in Spark ML or run PyTorch training, gradient descent is maximising the log-likelihood of your training data under the assumed distribution. The loss function choice = distribution assumption choice.

Loss = −log likelihood
MLE for LLMs
LLM pre-training maximises log-likelihood of next token: Σ log P(token_t | token_1...t-1). Cross-entropy loss on next-token prediction IS maximum likelihood over a categorical distribution. The entire GPT family is trained this way.

P(next token | context)
# MLE connection — loss functions are negative log-likelihoods import numpy as np # Gaussian MLE → MSE # Assume: yᵢ ~ Normal(ŷᵢ, σ²) # log P(y|ŷ) = -0.5*(y-ŷ)²/σ² - 0.5*log(2πσ²) # Maximise log-likelihood = minimise Σ(y-ŷ)² → MSE! def gaussian_nll(y_true, y_pred, sigma=1.0): return np.mean(0.5*((y_true-y_pred)/sigma)**2 + np.log(sigma)) # This is torch.nn.GaussianNLLLoss() # Bernoulli MLE → Binary Cross-Entropy # Assume: yᵢ ~ Bernoulli(p̂ᵢ) # log P(y|p) = y*log(p) + (1-y)*log(1-p) # Maximise → minimise -Σ[y*log(p)+(1-y)*log(1-p)] → BCE! def bernoulli_nll(y_true, p_pred): eps = 1e-9 return -np.mean(y_true*np.log(p_pred+eps)+(1-y_true)*np.log(1-p_pred+eps)) # Laplace MLE → MAE # Assume: yᵢ ~ Laplace(ŷᵢ, b) # log P(y|ŷ) = -|y-ŷ|/b - log(2b) # Maximise → minimise Σ|y-ŷ| → MAE! def laplace_nll(y_true, y_pred, b=1.0): return np.mean(np.abs(y_true-y_pred)/b + np.log(2*b)) # In Databricks — check your residuals to pick the right distribution residuals = y_true - y_pred print(f"Skewness: {residuals.skew():.3f}") # near 0 → Gaussian → MSE print(f"Kurtosis: {residuals.kurtosis():.3f}") # >3 → heavy tails → MAE/Huber
MAP — Maximum A Posteriori
MLE + prior belief = MAP. This is where regularisation comes from.
The big idea: MLE only uses the data. MAP says — also incorporate prior knowledge about what the weights should look like. A Gaussian prior on weights → L2 regularisation (Ridge). A Laplace prior → L1 regularisation (Lasso). Regularisation is not a trick — it is Bayesian reasoning.
MLE vs MAP
MLE: maximise P(data | w)
Find w that best explains the data. No assumptions about w itself.

MAP: maximise P(w | data) ∝ P(data | w) × P(w)
Find w that best explains data AND matches our prior belief about w.

log P(w|data) = log P(data|w) + log P(w)
= Likelihood term + Prior term
= Loss function + Regularisation term
Prior → Regularisation derivation
Gaussian prior: P(w) ∝ exp(−λ||w||²)
log P(w) = −λΣwᵢ²
MAP objective: minimise Loss + λΣwᵢ²
→ L2 regularisation (Ridge)!

Laplace prior: P(w) ∝ exp(−λ||w||₁)
log P(w) = −λΣ|wᵢ|
MAP objective: minimise Loss + λΣ|wᵢ|
→ L1 regularisation (Lasso)!

Regularisation strength λ = how strong the prior belief is.

Why Gaussian prior → L2?
A Gaussian prior says "I believe weights are close to 0 with bell-curve probability". Large weights are unlikely. This belief penalises large weights quadratically — exactly what L2 does. As λ→∞, all weights → 0.
Why Laplace prior → L1?
A Laplace prior has sharper peak at 0 and heavier tails than Gaussian. It more strongly believes weights should be exactly 0. This creates the sparsity property of L1 — some weights go to exactly 0 (feature selection).
Choosing λ in Databricks
In Spark ML: regParam=λ, elasticNetParam=0 (L2) or 1 (L1). Start with λ=0.01. If val loss improves → increase λ. If val loss worsens → decrease λ. Log every λ in MLflow.

Embeddings and RAG — norms in action
How vectors represent meaning and how cosine similarity powers retrieval
Connection: Embeddings are dense vectors trained by MLE (cross-entropy on prediction tasks). Stored in vector databases. Retrieved by cosine similarity (L2 norm based). The entire RAG pipeline is built on the math from Weeks 1–5.
What embeddings are
An embedding is a dense vector representation of any object (word, sentence, document, user, product) in a continuous vector space.

Semantically similar objects → nearby vectors (small cosine distance).
Semantically different objects → far vectors (large cosine distance).

Trained by MLE: predict context words (Word2Vec), predict next sentence (BERT), predict next token (GPT).

Azure OpenAI text-embedding-3-large → 3072-dimensional vectors.
How RAG uses norms
Index phase:
1. Chunk documents into passages
2. Embed each chunk: v = embed(chunk) ∈ ℝ^1536
3. Store in vector DB (Databricks Vector Search)

Query phase:
1. Embed query: q = embed(query) ∈ ℝ^1536
2. Compute cosine_sim(q, vᵢ) for all chunks
3. Return top-k chunks by similarity
4. Feed chunks + query to LLM

Cosine sim = dot(q,v) / (||q||₂ · ||v||₂)

SGD vs Momentum vs Adam — deep dive
How each optimizer works internally and when each shines
Connection to MLE: Optimizers are algorithms for performing gradient descent on the negative log-likelihood (loss). Different optimizers take the same gradient signal and apply different update rules — trading off speed, stability, and memory.

SGD update rule
w = w − α · ∇L
One fixed learning rate for all weights. Gradient directly drives the step.

Problem 1: Slow on flat loss surfaces (gradient is tiny).
Problem 2: Oscillates in ravines (large gradient in one direction, small in another).
Problem 3: Sensitive to learning rate choice.

Rarely used alone
Momentum update rule
v = β·v − α·∇L
w = w + v
Accumulates velocity v in the direction of consistent gradients. Dampens oscillations.

β=0.9: 90% previous velocity + 10% new gradient.
Like a ball rolling downhill — builds speed in consistent direction, slows in noisy direction.

Good for ravines
Adam update rule
m = β₁m + (1−β₁)∇L ← mean
v = β₂v + (1−β₂)∇L² ← variance
w = w − α·m̂/√(v̂+ε)
Per-parameter adaptive lr: large gradient → small step. Small gradient → large step.

β₁=0.9 (momentum), β₂=0.999 (variance), ε=1e-8 (stability).

Default for deep learning
LR Schedulers — reduce learning rate over time
Why use a scheduler?
Large lr at start = fast progress. Small lr later = fine-tuning near minimum. Fixed lr risks oscillating around minimum without converging.

Warmup + decay used in transformers: lr increases linearly for first 1000 steps (warmup), then decreases. Prevents destroying pre-trained weights at the start.

Always log lr as a metric in MLflow every epoch.
Common schedulers
StepLR: halve lr every N epochs. Simple.
CosineAnnealing: smooth cosine decay. Most common.
ReduceLROnPlateau: reduce when val loss stops improving. Most practical.
OneCycleLR: warmup then decay in one cycle. Fast convergence.
WarmupLinearDecay: transformer standard.

ReduceLROnPlateau is safest default
